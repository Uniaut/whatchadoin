Tempo (가칭)

출퇴근 시간 동안 주기적 Spotlight 팝업으로 "지금 무엇에 집중하고 있는가"를 기록하고, 작업별 focus·백그라운드 시간을 자동 측정하는 크로스플랫폼 데스크톱 생산성 앱.

1 제품 개요

한 줄 정의, 해결하려는 문제, 타깃 사용자, 핵심 가치 제안

한 줄 정의

Tempo는 칸반 보드로 업무를 관리하면서, 일과 시간 동안 주기적으로 "지금 무엇에 집중하고 있는지"를 가볍게 기록하게 하여, 작업별 실제 투입 시간(focus)과 위임/대기 시간(백그라운드)을 자동으로 정량화해 주는 데스크톱 앱이다.

해결하려는 문제

  • 주관적 시간 인식의 오차 — 사람은 자신이 무엇에 얼마나 시간을 썼는지 정확히 기억하지 못한다. 사후 타임시트 작성은 부정확하고 귀찮다.
  • 컨텍스트 스위칭의 비가시성 — 하루에 작업을 몇 번 바꿨는지, 한 작업에 얼마나 몰입했는지가 보이지 않는다.
  • AI 에이전트 위임 시간의 누락 — Claude Code 같은 에이전트가 백그라운드로 돌리는 동안의 "진척"은 기존 타이머 앱에서 측정 단위로 잡히지 않는다.
  • 회고 데이터 부재 — 하루를 마칠 때 "오늘 뭐 했지?"에 답할 구조화된 타임라인이 없다.

2 핵심 컨셉 정의

작업 상태 체계 · focus 단일성 규칙 · 시간 측정 로직

2.1 작업 상태 체계

작업(Task)은 항상 아래 다섯 상태 중 정확히 하나를 가진다. 보드의 칸(컬럼)은 이 상태와 1:1 매핑된다. 단, 진행중·focus와 진행중·백그라운드는 같은 "진행중" 컬럼 안에서 배지로 구분한다(컬럼 4개, 상태 5개).

상태의미시간 측정focus 점유
해야할일 TODO착수 전 백로그측정 안 함불가
펜딩 PENDING외부 의존·블로커로 대기(리뷰 대기, 답변 대기 등). 일시적으로 진행 불가측정 안 함불가
진행중·focus지금 내가 직접 집중하는 단 하나의 작업focus 시간 누적점유(동시 1개)
진행중·백그라운드진행 중이나 내 직접 집중은 아님(에이전트 위임, 빌드/렌더 대기 등)백그라운드 시간 누적비점유(동시 다수 허용)
완료 DONE종료된 작업측정 중단·확정불가

2.2 'focus는 단 하나' 규칙과 전환 동작

시스템 전체에서 진행중·focus 작업은 항상 0개 또는 1개다. 사용자가 작업 B를 focus로 지정하면 다음이 원자적으로(atomic) 일어난다:

  1. 현재 focus였던 작업 A의 진행 중 focus 세션을 종료(타이머 stop, 종료시각 기록).
  2. 작업 A를 어디로 보낼지 강등 규칙에 따라 결정:
    • A가 여전히 "진행 중"이어야 하면 → 백그라운드로 강등하고 백그라운드 세션 시작.
    • 사용자가 명시적으로 "A는 잠시 멈춤"을 선택하면 → 펜딩 또는 TODO로 이동, 어떤 타이머도 돌지 않음.
  3. 작업 B의 focus 세션 시작(타이머 start). B가 백그라운드였다면 백그라운드 세션을 종료하고 focus 세션으로 승격.
기본값(가정). 별도 선택 없이 다른 작업을 focus로 만들면, 직전 focus 작업은 자동으로 백그라운드로 강등된다(흔한 컨텍스트 스위칭 = 잠깐 자리 비움으로 간주). 단, Spotlight 팝업(§4)에서는 "이전 작업을 멈출까요/계속 둘까요?"를 1탭으로 물어 오분류를 줄인다.

2.3 focus 시간 vs 백그라운드 시간

구분focus 시간백그라운드 시간
정의사용자가 직접 그 작업에 몰입한 시간작업이 진행되지만 사용자 집중 밖에서 흐른 시간(에이전트 실행, 빌드 등)
동시성전체 합산 시 겹치지 않음(항상 1개)여러 작업이 동시 누적 가능
누적 시작작업이 focus로 진입한 순간작업이 백그라운드로 진입한 순간
누적 중단다른 작업 focus 전환 / 일과 종료 / 유휴(idle) 임계 초과 / 펜딩·완료 이동작업이 펜딩·완료·TODO로 이동 / 일과 종료
대표 지표몰입도·실투입위임/병렬 효율
핵심 불변식. 임의의 시점 t에서 focus 상태 작업 수 ∈ {0,1}. 따라서 하루 focus 시간 합 ≤ 일과 총 길이. 백그라운드 시간 합은 일과 길이를 초과할 수 있다(병렬).

3 사용자 시나리오 (User Story)

하루 흐름: 출근 → 작업 전환 → 에이전트 위임 → 퇴근 → 회고

US-1 · 출근, 일과 시작

개발자 지민은 09:00에 Tempo를 연다. 어제 설정한 일과(09:00–18:00)가 자동 시작되며 첫 Spotlight 팝업이 뜬다. "오늘 첫 focus는?"에 보드의 결제 모듈 리팩터링을 선택 → 즉시 focus 타이머가 돈다.
가치: 하루의 시작점과 첫 집중 대상이 1초 만에 기록된다.

US-2 · 작업 전환 (컨텍스트 스위칭)

10:20, 동료가 코드리뷰를 요청한다. 지민은 보드에서 PR #482 리뷰 카드를 focus로 드래그한다. 시스템은 결제 모듈 focus 세션을 종료하고 자동으로 백그라운드로 강등(혹은 펜딩 선택). 5분 뒤 Spotlight가 "여전히 PR 리뷰 중?"을 물으면 "맞음 — 코멘트 3개 작성 중"이라고 한 줄 남긴다.
가치: 전환 횟수·각 작업 점유 시간이 자동으로 기록된다.

US-3 · 에이전트에게 위임 (백그라운드)

11:00, 지민은 Claude Code에게 테스트 커버리지 90% 작업을 맡긴다. 이 카드를 진행중·백그라운드로 지정하고, 본인은 결제 모듈을 다시 focus로 가져온다. 이제 두 타이머가 동시에 흐른다 — 결제 모듈은 focus, 테스트 작업은 백그라운드.
가치: 내가 직접 한 시간과 에이전트가 대신 진행한 시간이 분리 집계된다.

US-4 · 점심·자리비움 (유휴 처리)

12:30, 지민이 점심으로 자리를 비운다. 키보드/마우스 입력이 없어 5분 후 유휴 임계를 넘기면 focus 타이머가 자동 일시정지되고 복귀 시 "12:30–13:25 자리 비움이었나요? 이 시간을 focus에서 제외할까요?"를 묻는다.
가치: 부풀려진 집중 시간이 자동 보정된다.

US-5 · 퇴근, 회고

18:00, 지민이 "일과 종료"를 누른다. 모든 타이머가 멈추고 일과 리포트가 뜬다 — 작업별 focus/백그라운드 시간, 전환 횟수, 그리고 각 Spotlight 때 남긴 한 줄 기록이 시간순 타임라인으로 정리된다. 지민은 "결제 모듈 3h12m, 전환 7회"를 보며 내일 우선순위를 정한다.
가치: 사후 작성 없이 구조화된 하루 회고가 완성된다.

그림 3. 데이터 모델 관계도(ER)

엔티티 설명

  • Task — 작업 단위. 현재 상태는 단일 컬럼, 누적 시간은 focus_total_sec/bg_total_sec에 캐시(원천은 TimeSession 합).
  • TimeSession — 시간 측정의 원천(source of truth). 하나의 (작업 × 종류 × 연속 구간). ended_at=null이면 현재 진행 중. focus 세션은 전체에서 동시 1개만 진행 상태일 수 있다(불변식).
  • PopupLog — 각 Spotlight 노출/응답 1건. 타임라인과 회고의 원재료. outcome로 무응답·스누즈·신규생성 구분.
  • Workday — 출퇴근 1일 단위. 팝업 스케줄과 리포트 집계의 경계.
  • Project(선택) — 작업 그룹핑·색상. MVP에서는 단순 태그 수준, 확장 시 청구 단위.
  • Settings — 단일 레코드(앱 전역 설정).
가정. 시간은 모두 UTC로 저장하고 표시 시 로컬 타임존 변환. 자정 넘김은 §7-EC3 규칙으로 세션을 분할 저장한다.

7 엣지 케이스

정상 흐름에서 벗어나는 상황과 처리 규칙

ID상황처리 규칙
EC1일과 시간 외에 작업일과 종료 후엔 팝업이 뜨지 않고 신규 측정도 기본 비활성. 사용자가 작업을 focus로 옮기면 "지금 일과를 다시 열까요?(연장/임시 일과)"를 제안. 임시 일과로 측정하면 리포트에 '일과 외' 구간으로 별도 표기.
EC2팝업 무응답표시 후 타임아웃(기본 60초) 시 자동 닫힘 + PopupLog(MISSED). focus는 직전 상태 유지(시간 계속 누적). 연속 N회(기본 3) 무응답 시 "자리 비움?"으로 간주해 다음 팝업 주기를 늘리고 유휴 검사 강화.
EC3자정 넘김진행 중 세션을 23:59:59에서 split하여 당일/익일 2개 세션으로 저장(집계 경계 보존). 일과가 자정을 넘기게 설정된 경우(야근/야간 근무) Workday는 시작일 기준 1건으로 유지하되 날짜 경계 마커를 둠.
EC4일과 종료 미클릭(잊고 자리 뜸)설정 종료시각 + 유예(기본 2h) 경과 & 무활동이면 자동 마감. 마감 시각은 '마지막 활동 시각'으로 보정(빈 시간 부풀림 방지). 다음 실행 시 "어제 일과를 OO시에 마감했어요" 안내.
EC5동시 백그라운드 다수허용(설계 의도). 단 카드별 독립 누적이라 총 백그라운드 합 > 실제 경과가 정상. 리포트에서는 'wall-clock(실경과)'과 '백그라운드 누적합'을 분리 표기해 오해 방지. 상한 초과 시 경고.
EC6focus 작업을 직접 완료/펜딩 이동focus 세션 종료 후 focus 공석(0개). 다음 Spotlight 또는 즉시 프롬프트로 새 focus 유도. 공석 상태에서는 focus 시간 누적 없음.
EC7시스템 슬립/절전 복귀monotonic이 아닌 wall-clock 기준 재계산. 슬립 구간은 focus에서 제외(유휴로 처리), 단 BG가 '에이전트 실행'이면 사용자 선택으로 포함 가능.
EC8시계 변경/타임존 이동저장은 UTC. 사용자 시계 역행 감지 시 음수 경과를 0으로 클램프하고 이상 로그 기록.
EC9앱 강제 종료 중 진행 세션세션 시작은 영속 저장되어 있으므로 재실행 시 미마감 세션을 감지 → '마지막 하트비트 시각'까지로 복구 마감 후 사용자에게 보정 제안.
EC10유휴 권한 거부(특히 Linux/Wayland)OS 유휴 API 불가 시 수동 모드로 폴백: 팝업 응답·명시적 일시정지에만 의존. 설정에서 상태 안내.

9.1 크로스플랫폼 프레임워크

후보장점단점본 제품 적합성
Tauri (Rust + WebView)
권장		경량(수~십 MB), 메모리 적음, OS 네이티브 API·트레이·전역 단축키·유휴 감지에 강함, 보안 모델 우수Rust 학습 곡선, 일부 WebView 렌더 편차높음 — 상주형 트레이 앱 + 빈번한 OS 이벤트(유휴/입력)와 궁합
Electron (Chromium + Node)생태계·자료 풍부, 단일 렌더 엔진으로 일관성, 빠른 개발무겁고(100MB+) 메모리 큼 — 종일 상주 앱엔 부담중 — 빠른 프로토타입엔 좋으나 상주 부담
Flutter Desktop단일 코드·고성능 렌더, 일관 UI데스크톱 OS 통합(트레이/유휴) 성숙도 낮음중하 — OS 통합 의존도 높은 본 제품엔 리스크
권장: Tauri. 본 앱은 하루 종일 백그라운드 상주하며 트레이 아이콘, 전역 단축키(빠른 체크인), OS 유휴/입력 감지, 화면 중앙 Spotlight 모달을 자주 호출한다. 이런 "가벼운 상주 + 강한 OS 통합" 요건에 Tauri의 저자원·네이티브 접근이 가장 부합한다. 빠른 검증이 최우선이면 Electron으로 PoC 후 Tauri 이관도 합리적이다.

9.2 아키텍처 & 데이터 저장

  • 프론트엔드(WebView): TypeScript + React, 보드는 dnd-kit/@hello-pangea/dnd, 차트는 경량 라이브러리. 상태관리 Zustand.
  • 코어(Rust): 시간 측정 엔진·스케줄러·유휴 감지·트레이를 네이티브에 둠(UI가 닫혀도 정확한 측정 보장). 1초 틱 대신 이벤트 기반 + wall-clock 차감으로 절전.
  • 데이터 저장: 로컬 SQLite(local-first). 마이그레이션 관리, 트랜잭션으로 focus 전환 원자성 보장. 파일은 OS 앱 데이터 디렉터리에 저장, 사용자 내보내기(JSON/CSV) 제공.
  • 동기화(확장): 초기엔 미포함. 확장 시 SQLite → 동기 가능 백엔드(예: 서버 + CRDT/이벤트 로그)로 점진 이행.
  • 시간 정합성: 모든 시각 UTC 저장, monotonic 보조 + wall-clock 기준 재동기화(슬립/시계변경 대비, §7).
